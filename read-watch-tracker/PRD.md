# PRD — 個人閱讀與觀影紀錄系統

## 1. 應用說明

現代人的閱讀與觀影習慣日趨分散。書籍可能來自實體書店、電子書平台，電影可能散落於串流服務、電影院或影碟收藏。課程與文獻也常分散於 MOOC、論文網站、筆記工具與瀏覽器書籤。

本專案目標是設計一個簡潔的 Web App，讓使用者能統一記錄、管理並回顧自己的閱讀、觀影、課程與文獻紀錄。

### 1.1 核心問題

- 影音、書籍、課程與文獻平台分散，難以統一追蹤進度。
- 心得筆記與評分缺乏集中管理，後續回顧成本高。
- 使用者需要以低成本保存清單、標籤、狀態與備註。

### 1.2 開發動機

- 打造跨媒介的單一紀錄平台。
- 降低個人閱讀、觀影與學習紀錄的管理成本。
- 讓使用者能用列表、篩選、標籤與儀表板回顧自己的文化消費歷程。

### 1.3 目標使用者

- 喜愛閱讀與觀影，需記錄進度與評分的愛好者。
- 需要追蹤課程、文獻與實作參考價值的學生或研究者。
- 想用單一 Web App 管理個人收藏與心得的人。

## 2. 核心功能

### 2.1 搜尋：資料檢索

- 支援關鍵字搜尋標題。
- 支援依標籤、類型與進度篩選。
- 使用者可快速找到下一筆要閱讀、觀影、學習或整理的項目。

### 2.2 收藏：新增紀錄

- 新增、編輯、刪除書籍或影片項目。
- 支援課程與文獻/論文紀錄。
- 每筆紀錄可設定標題、類型、題材、標籤與備註。

### 2.3 追蹤：狀態管理

- 支援想讀/想看、閱讀中/觀影中/實作中、完成等狀態。
- 清單需一眼看出目前進度。
- 頁數、片長、課程進度百分比可列為後續擴充。

### 2.4 筆記：評價分類

- 支援 1~5 星評分。
- 支援多重標籤分類。
- 支援心得、重點摘要與實作參考價值備註。

## 3. 系統架構

### 3.1 前端

- 使用 HTML、CSS、JavaScript。
- 使用者以一般瀏覽器操作，不需安裝桌面程式。
- 負責畫面呈現、表單驗證、清單互動與 HTTP 請求。

### 3.2 後端

- 正式目標使用 FastAPI + Uvicorn。
- 目前 Demo 先提供標準庫 ASGI app，可直接由 Uvicorn 啟動。
- 承載商業邏輯與 REST 路由。
- 回應格式為 JSON。
- 若後續改成 FastAPI app，可再使用 FastAPI 自動文件輔助測試與驗收。

### 3.3 資料庫

- 正式目標使用 PostgreSQL。
- 目前 Demo 使用 SQLite 持久化，PostgreSQL 初始化 schema 見 `schema.sql`。
- 持久化儲存清單、進度、評分、標籤、心得與參考連結。
- 以 schema 與資料表設計維護一致性。

### 3.4 資料傳輸

- 前後端只透過 REST API 交換資料。
- 請求與回應格式採 JSON。
- 系統型態為單體後端 + 關聯式資料庫。

### 3.5 不納入本階段架構

- API Gateway。
- Redis。
- 外部書籍、電影、串流或文獻 API。
- 多使用者權限與社群分享。

## 5. 功能邊界

### 5.1 In Scope

- 單一使用者清單管理。
- 書籍、影片、課程、文獻/論文 CRUD。
- 標題關鍵字搜尋。
- 類型、標籤、題材與狀態篩選。
- 1~5 星評分與心得備註。
- PostgreSQL 持久化。
- FastAPI REST API。
- 純 HTML/CSS/JavaScript 前端。

### 5.2 Out of Scope

- 外部書籍、電影、串流、課程或論文 API 串接。
- 多使用者登入、權限與共享。
- 推薦演算法。
- AI 自動摘要或自動分類。
- API Gateway、Redis、微服務架構。
- 手機原生 App。

## 6. 預期成果

- **跨媒介整合系統**：建立統一 Web App，整合書籍、影視、課程與文獻紀錄。
- **直觀的進度管理**：提供儀表板與卡片式介面，追蹤想讀/想看、進行中與已完成狀態。
- **深度心得紀錄**：建立評分與標籤系統，支援詳細心得筆記。
- **高效能技術實作**：透過 RESTful API 與資料庫持久化，確保資料存取穩定並提供響應式體驗。

## 7. KPI（可量測，同步 `slo.config.json`）

- `p95_latency_ms <= 120.0`
- `timeout_rate <= 0.002`
- `cost_per_request <= 0.2`
- `quality_drop_ratio <= 0.05`
- `error_rate <= 0.01`
- 相對門檻：p95 <= baseline x 0.75；cost <= baseline x 0.7

## 8. NO-GO 條件

- 任一核心測試失敗。
- 任一 KPI 超標。
- 觀測資料缺失（artifact 不完整）。
- CRUD 任一核心流程不可用。
- PostgreSQL schema 與 API 欄位不一致。
- 前端無法透過 REST API 完成新增、查詢、修改或刪除。

## 9. 風險與假設

- **風險**：資料表過度分散、欄位命名不一致、狀態值不一致、前端與 API schema 漂移、PostgreSQL 連線設定不穩。
- **假設**：本階段以單一使用者為主；資料量以個人收藏規模估計；不需處理外部 API 限流與資料授權問題。
- **緩解策略**：先定義資料模型與狀態 enum；用測試覆蓋 CRUD、搜尋與篩選；以 schema migration 或初始化 SQL 固定資料表欄位。
